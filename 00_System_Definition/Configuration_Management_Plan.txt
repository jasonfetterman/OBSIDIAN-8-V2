------------------------------------------------------------
1. PURPOSE
------------------------------------------------------------

Define configuration control, version management,
change control procedures, and baseline governance
for OBSIDIAN‑8 V3.

Configuration management ensures:
- Traceability
- Reproducibility
- Controlled evolution
- Audit readiness

This plan operates in conjunction with the
Quality_Management_Plan.txt and
Verification_Traceability_Matrix.txt
to preserve system integrity across the full lifecycle.

------------------------------------------------------------
2. CONFIGURATION BASELINES
------------------------------------------------------------

The following configuration baselines are defined:

2.1 System Baseline  
Complete document set and architecture state.

2.2 Hardware Baseline  
Frame revision  
Actuator models  
Battery chemistry  
BMS revision  
Power distribution revision  

2.3 Software Baseline  
ROS 2 package versions  
Message definitions version  
Firmware versions  
Launch configurations  

2.4 Production Baseline  
BOM revision  
Assembly instructions  
Calibration procedures  

No baseline may change without formal approval.

Baseline repository map:

| Baseline | Repository / Path | Record File |
|-----------|------------------|--------------|
| System    | /00_System_Definition/ | system_manifest.yaml |
| Hardware  | /02_Electrical/, /01_Mechanical/ | hardware_baseline_v3.2.yaml |
| Software  | /11_Codebase/obsidian_ws/ | software_manifest_v3.2.yaml |
| Production| /12_STL_Production/00_Master_List/ | production_baseline.csv |

------------------------------------------------------------
3. VERSION IDENTIFICATION
------------------------------------------------------------

Version format:  V[Major].[Minor].[Revision]

Major — Architecture‑level change  
Minor — Feature addition or improvement  
Revision — Documentation correction or minor update  

All controlled files must display the version identifier.

------------------------------------------------------------
4. CHANGE CONTROL PROCESS
------------------------------------------------------------

4.1 Change Request (CR)

Any modification must begin with a documented CR including:
- Description
- Justification
- Affected subsystems
- Risk assessment
- Impact analysis

All CRs are logged under:
  /10_Lifecycle/Change_Requests/
and numbered sequentially as:
  CR‑YYYY‑###  (e.g., CR‑2024‑012)

4.2 Classification

Class A – Safety or structural impact  
Class B – Performance affecting  
Class C – Documentation only  

Class A changes require full regression testing before approval.

------------------------------------------------------------
5. APPROVAL AUTHORITY
------------------------------------------------------------

Required approvals by change class:

| Class | Approvers |
|:------|:-----------|
| A | System Architect + Safety Reviewer + Production Manager |
| B | System Architect + Production Manager |
| C | System Architect |

No unilateral changes are permitted.

------------------------------------------------------------
6. CONFIGURATION RECORDS
------------------------------------------------------------

Each production unit must record:
- Serial number
- Hardware revision
- Software version
- Calibration version
- Battery configuration
- Deployment profile

Records must be retained for no less than 5 years.

------------------------------------------------------------
7. SOFTWARE CONTROL
------------------------------------------------------------

Source code must be:
- Version‑controlled (Git)
- Tagged per release
- Linked to SRS revision
- Linked to test validation record

No production deployment may originate from an untagged commit.

------------------------------------------------------------
8. DOCUMENT CONTROL
------------------------------------------------------------

All engineering documents must:
- Include revision number
- Include status (Draft / Review / Authoritative)
- Be archived on update
- Maintain change history

Uncontrolled copies are prohibited.

------------------------------------------------------------
9. CONFIGURATION AUDITS
------------------------------------------------------------

Periodic audits must verify:
- Software version matches baseline
- Hardware matches BOM revision
- Calibration records complete
- Safety architecture intact

Audit frequency: every 6 months or before:
- Phase 2 exit
- Scaled production release
- Field deployment

------------------------------------------------------------
10. FIELD CHANGE MANAGEMENT
------------------------------------------------------------

Field modifications require:
- Documented CR
- Risk evaluation
- Verification testing
- Baseline update
- Registry update

Unauthorized field modification is prohibited.

------------------------------------------------------------
11. END‑OF‑LIFE CONTROL
------------------------------------------------------------

When a unit is decommissioned:
- Configuration record archived
- Serial marked inactive
- Firmware version recorded

No decommissioned unit may re‑enter the fleet
without complete requalification.

------------------------------------------------------------
12. DESIGN INTENT
------------------------------------------------------------

Configuration management supports:
- Predictable system evolution
- Safety preservation across revisions
- Traceable engineering changes
- Audit‑ready documentation
- Production consistency

Controlled change preserves system integrity and safety.
